## Findings

### [IMPORTANT] XSS - single quote breakout dans onclick avec donnees utilisateur
- **Fichier** : app.html:3014, app.html:3906
- **Description** : `escHtml()` (ligne 5082) utilise `div.textContent/div.innerHTML` qui echappe `< > &` mais PAS les apostrophes (`'`). Or les attributs onclick utilisent des single quotes pour delimiter les arguments string :
  - Ligne 3014 : `onclick="dashboardCardClick('${escHtml(item.name)}')"` - un nom d'operation contenant `'` (ex: "Filtre d'habitacle", frequent en francais) casse la syntaxe JS. Un nom malveillant comme `x');alert(document.cookie);//` permet l'execution de code arbitraire.
  - Ligne 3906 : `onclick="window._stationChartsOpen.has('${escHtml(station)}')"` - meme probleme avec les noms de station (ex: "Total l'Esplanade").
  - Les identifiants vehicules dans onclick (lignes 2436, 5184, 5547) utilisent `v.id` sans echappement. Les IDs generes par l'app sont alphanumeriques (safe), mais un import JSON pourrait contenir des IDs arbitraires.
- **Risque** : Execution de JS arbitraire dans le contexte de l'application. Vecteur realiste : nom d'operation contenant une apostrophe (courant en francais), ou import de JSON malveillant.
- **Correction** : Creer une fonction `escAttr(s)` qui echappe aussi les quotes :
  ```javascript
  function escAttr(s) { return escHtml(s).replace(/'/g, '&#39;').replace(/"/g, '&quot;'); }
  ```
  Utiliser `escAttr()` dans tous les onclick passant des strings. Idealement, migrer vers `data-*` attributes + event delegation pour eliminer les onclick inline.
- **Statut** : [x] corrige - escAttr() ajoutee, utilisee dans onclick dashboard + station

### [IMPORTANT] XSS - schedule name non echappe dans projections km
- **Fichier** : app.html:2590, app.html:2606
- **Description** : Dans `renderKmTab()`, le nom de l'operation schedule (`s.name`) est insere dans `m.label` sans echappement (ligne 2590 : `` `${nextKm.toLocaleString()} km - ${s.name}` ``), puis `m.label` est injecte dans innerHTML (ligne 2606 : `${m.label}`). Vecteur : import JSON avec un schedule name contenant du HTML, ou reponse IA Groq manipulee.
- **Risque** : Injection HTML/JS via le nom d'une operation de maintenance.
- **Correction** : Echapper `s.name` a la construction du label :
  ```javascript
  milestones.push({ label: `${nextKm.toLocaleString()} km - ${escHtml(s.name)}`, km: nextKm });
  ```
- **Statut** : [x] corrige - escHtml(s.name) dans milestones

### [IMPORTANT] Firestore rules shares - any auth user can delete any share
- **Fichier** : firestore.rules:8-14
- **Description** : La regle actuelle pour les shares est :
  ```
  allow create: if request.auth != null && request.resource.data.createdBy == request.auth.uid;
  allow delete: if request.auth != null && resource.data.createdBy == request.auth.uid;
  ```
  La regle delete verifie bien `resource.data.createdBy == request.auth.uid` - ceci est CORRECT. Le delete est restreint au createur du share.
  **Mise a jour** : En relisant attentivement les rules actuelles, cette regle EST correcte. Le finding precedent etait errone - la condition `resource.data.createdBy == request.auth.uid` sur le delete restreint bien la suppression au createur. Finding ferme.
- **Statut** : [x] corrige (finding precedent errone - rules deja correctes)

### [IMPORTANT] XSS - v.name non echappe dans title du carnet HTML exporte
- **Fichier** : app.html:4755
- **Description** : `<title>Mon Garage - ${v.name}</title>` sans `escHtml()`. La balise `<title>` est un element raw text - une valeur comme `</title><script>alert(1)</script>` casserait la balise et injecterait du JS dans le fichier HTML exporte. Le fichier est ouvert depuis une URL `blob:` (isolee du domaine de l'app), donc l'impact est limite au contexte du blob. Cela reste une mauvaise pratique.
  Note : le dossier de vente (ligne 4933) utilise correctement `escHtml(v.name)` dans son title.
- **Risque** : Self-XSS dans le fichier HTML exporte. Impact faible (contexte blob isole).
- **Correction** : `<title>Mon Garage - ${escHtml(v.name)}</title>`
- **Statut** : [x] corrige - escHtml(v.name) dans title export

### [RECOMMANDE] escHtml insuffisant pour les attributs HTML (double quotes)
- **Fichier** : app.html:2922, app.html:2863
- **Description** : `escHtml()` n'echappe pas les double quotes (`"`). Or sur la ligne 2922, `title="${escHtml(a.detail)}"` insere le detail d'alerte dans un attribut HTML delimite par des double quotes. Le champ `a.detail` contient `item.name` non echappe (ligne 2863 : `` `${item.name} prevu a...` ``). Si `item.name` contient un `"`, cela casse l'attribut title et permet une injection d'attribut HTML (ex: `" onmouseover="alert(1)`).
  L'impact est mitige par le fait que le contenu text est escape (`<` et `>` sont filtres), mais une injection d'event handler reste possible via la quote non echappee.
- **Correction** : Utiliser `escAttr()` (qui echappe aussi `'` et `"`) pour tous les contextes d'attributs HTML, ou echapper `item.name` dans le detail :
  ```javascript
  detail: `${escHtml(item.name)} prevu a ${s.nextKm.toLocaleString()} km...`
  ```
- **Statut** : [x] corrige - escAttr(item.name) dans detail + escAttr(a.detail) dans title

### [RECOMMANDE] CSP manque base-uri et form-action
- **Fichier** : firebase.json:22
- **Description** : La CSP ne definit pas `base-uri` ni `form-action`. Combinee a `unsafe-inline`, l'absence de `base-uri` permet une injection de `<base href>` qui redirigerait les URLs relatives. L'absence de `form-action` permettrait un formulaire de soumission vers un domaine externe.
- **Correction** : Ajouter `base-uri 'self'; form-action 'self';` a la CSP.
- **Statut** : [x] corrige - base-uri et form-action ajoutes dans firebase.json

### [RECOMMANDE] CSP unsafe-inline toujours present
- **Fichier** : firebase.json:22
- **Description** : `script-src 'unsafe-inline'` est necessaire pour l'app single-file mais annule la protection XSS de la CSP. Les findings XSS ci-dessus seraient bloques par une CSP stricte avec nonces ou hashes.
- **Statut** : [ ] ouvert (acceptable pour le contexte single-file sans build system)

### [RECOMMANDE] config/api_keys lisible par tout utilisateur authentifie
- **Fichier** : firestore.rules:15-16
- **Description** : La regle `allow read: if request.auth != null` sur `config/{doc}` permet a tout utilisateur authentifie dans le projet Firebase de lire les cles API tierces (Groq, RapidAPI). En v3 self-hosted chaque user a son propre projet, donc cela ne concerne que les co-utilisateurs du meme projet Firebase.
- **Statut** : [ ] ouvert (acceptable si mono-utilisateur par projet Firebase)

### [RECOMMANDE] SW sans validation integrite des reponses cachees
- **Fichier** : sw.js:40-41
- **Description** : Le service worker cache toute reponse `res.ok` sans verifier Content-Type ni integrite. Risque theorique de cache poisoning en cas de MITM (improbable sur HTTPS Firebase Hosting). Le SW utilise correctement network-first avec fallback cache.
- **Statut** : [ ] ouvert

## Points conformes

### Nouvelles verifications v3

- **Setup screen** : les champs du formulaire Firebase config sont lus avec `.value.trim()` et passes a `firebase.initializeApp()` qui valide internement. Pas d'injection possible via les champs de setup.
- **IndexedDB storage config** : IndexedDB est same-origin, pas de risque cross-origin. Les valeurs stockees sont uniquement relues comme config Firebase, pas injectees dans du HTML.
- **Boot sequence** : la cascade IndexedDB > config.js > setup est lineaire et deterministe (`async function boot()` avec early return). Pas de race condition possible.
- **resetFirebaseConfig()** : protege par `confirm()` + recharge la page. Pas d'abus possible au-dela du DoS volontaire (l'user doit confirmer).
- **Suppression ALLOWED_UIDS** : pas de regression. En v3, `garages/{userId}` avec `request.auth.uid == userId` isole chaque utilisateur a son propre document. Plus besoin d'une whitelist.
- **Firestore rules shares** : les regles delete verifient correctement `resource.data.createdBy == request.auth.uid`. Seul le createur peut supprimer ses shares.
- **SW config.js** : le service worker ne cache plus `config.js` dans sa liste ASSETS (corrige depuis le precedent audit).

### Auth et sessions

- Firebase Auth via Google Sign-In : session geree par Firebase SDK, tokens Firebase Auth standards, pas de gestion manuelle de session.
- `onAuthStateChanged` correctement utilise pour gerer les transitions login/logout.
- `GARAGE_DOC = 'garages/' + user.uid` : le path du document est derive de l'UID Firebase authentifie, pas de saisie utilisateur.
- `googleAccessToken` stocke en memoire uniquement (pas persiste dans localStorage/IndexedDB).
- Share view (`isShareView`) correctement isolee : pas d'acces auth, lecture seule du document share, escHtml sur vehicleName/vehicleModel/ops.

### XSS / Injection

- `escHtml()` correctement utilisee sur 65 occurrences pour les contenus texte dans innerHTML.
- `sanitizeCsvCell()` applique sur tous les champs texte du CSV (protection injection CSV).
- Import JSON : validation de structure complete (types, arrays, `ops.every(o => typeof o === 'string')`).
- Pas d'`eval()`, pas de `Function()`, pas de `document.write()`, pas d'`insertAdjacentHTML`.
- `textContent` utilise pour les messages d'erreur et OCR (loginError, setupError, ocrStatus).
- Share view : donnees limitees (date + ops seulement), escHtml applique.
- OCR output traite comme texte brut, affiche via `textContent`, pas de risque d'injection.
- Groq AI response : parsee en JSON, validee (name string, kmInterval/yearInterval numbers), names affichees avec escHtml dans les renderers (sauf le bug projection km cite ci-dessus).

### Reseau et transport

- HSTS : `max-age=31536000; includeSubDomains` - correct.
- X-Frame-Options : `DENY` - empeche le clickjacking.
- X-Content-Type-Options : `nosniff` - empeche le MIME sniffing.
- Referrer-Policy : `strict-origin-when-cross-origin` - politique restrictive.
- Permissions-Policy : `camera=(self), microphone=()` - micro bloque, camera limitee au meme origin.
- CSP connect-src : whitelist stricte des domaines Firebase, Google APIs, Groq, RapidAPI. Pas de wildcard.
- `window.open()` avec validation URL prefix (drive.google.com, firebasestorage uniquement).

### Crypto et tokens

- `generateToken()` utilise `crypto.getRandomValues(new Uint8Array(16))` = 128 bits d'entropie. Robuste pour des tokens de partage a expiration.
- `genId()` : IDs numeriques collision-safe (Date.now()*1000 + compteur). Pas usage crypto mais adequat pour des IDs internes.

### Configuration et secrets

- `config.js` dans `.gitignore` - pas de commit de config Firebase.
- `config.example.js` avec placeholders uniquement (`VOTRE_API_KEY`, etc.), pas de vrais secrets.
- Cles API tierces stockees dans Firestore `config/api_keys` (protegees par auth rules), pas dans le code.
- Pas de secret, token ou credential hardcode dans app.html.
- `PLATE_API_TOKEN` et `GROQ_API_KEY` envoyes uniquement dans les headers des requetes aux APIs respectives (pas dans les URLs, pas en query params).

### Service Worker

- Network-first strategy avec fallback cache : les mises a jour sont toujours preferees.
- APIs Firebase, Google, CDNs correctement exclues du cache (ligne 28-34 de sw.js).
- `skipWaiting()` + `clients.claim()` : activation immediate du nouveau SW.
- Nettoyage des anciens caches dans l'event activate.

### Notifications

- Truncature `.slice(0, 80)` / `.slice(0, 120)` sur title/body pour eviter des payloads excessifs.
- Notification API est text-only, pas d'interpretation HTML.
