## Findings

### [IMPORTANT] Fonctions mortes : getGarageDoc() et isThermal() jamais appelees
- **Fichier** : `app.html:1424`, `app.html:1792`
- **Description** : `getGarageDoc()` est definie mais jamais appelee - le code utilise directement `GARAGE_DOC`. `isThermal()` est definie mais jamais appelee non plus. Ce sont des reliquats qui alourdissent le code et pretent a confusion.
- **Correction** : Supprimer `getGarageDoc()` si non utilisee en dehors du fichier. Supprimer `isThermal()` ou l'utiliser dans les endroits ou `!isElectric(v) && !isHybrid(v)` est ecrit en dur.
- **Statut** : [x] corrige - getGarageDoc() supprimee, isThermal() conservee (helper API documente)

### [IMPORTANT] sw.js pre-cache config.js qui peut ne pas exister en v3
- **Fichier** : `sw.js:4`
- **Description** : Le Service Worker inclut `'./config.js'` dans la liste `ASSETS` a pre-cacher. En v3, config.js est optionnel (la config peut etre saisie dans l'app via IndexedDB). Si config.js n'existe pas, `caches.addAll()` echoue et le SW ne s'installe pas, cassant le mode offline.
- **Correction** : Retirer `'./config.js'` de la liste `ASSETS` dans sw.js. Le fichier est deja charge avec `onerror=""` dans app.html, donc son absence est geree cote HTML. Alternativement, cacher chaque asset individuellement avec un try/catch pour tolerer les 404.
- **Statut** : [x] corrige - config.js retire de ASSETS dans sw.js

### [IMPORTANT] renderAll et renderAllSections : duplication du code de gestion des search bars
- **Fichier** : `app.html:5122-5155`
- **Description** : Les deux fonctions `renderAll()` et `renderAllSections()` dupliquent exactement le meme bloc de code pour afficher/cacher les search bars (historySearchBar, fuelSearchBar, chargeSearchBar). De plus, `renderAllSections()` appelle `updateNavForVehicleType()` a la fin alors que cette fonction est deja dans `_commonRenderers` et donc deja executee dans la boucle - double appel.
- **Correction** : Extraire le bloc search bars dans une fonction utilitaire appelee par les deux. Retirer le `updateNavForVehicleType()` redondant a la fin de `renderAllSections()`.
- **Statut** : [x] corrige - updateSearchBars() extraite, double appel updateNavForVehicleType() supprime

### [IMPORTANT] Firestore rules shares : tout utilisateur authentifie peut creer/supprimer n'importe quel share
- **Fichier** : `firestore.rules:9-11`
- **Description** : La regle `allow create, delete: if request.auth != null;` autorise n'importe quel utilisateur authentifie a supprimer les shares d'un autre utilisateur (il suffit de connaitre le token). Le code JS verifie `createdBy == currentUser.uid` cote client, mais la regle Firestore ne le fait pas cote serveur.
- **Correction** : Restreindre le delete au createur : `allow delete: if request.auth != null && resource.data.createdBy == request.auth.uid;`. Pour le create, ajouter une validation que `request.resource.data.createdBy == request.auth.uid`.
- **Statut** : [x] corrige - rules shares restreintes au createur (create + delete)

### [IMPORTANT] Boot sequence : pas de feedback si IndexedDB echoue silencieusement
- **Fichier** : `app.html:31-40`
- **Description** : `loadFirebaseConfig()` catch silencieusement toute erreur IndexedDB et retourne `null`. Si IndexedDB est bloque (mode prive Safari, quota), l'app tombe en fallback config.js puis setup screen sans expliquer pourquoi une config precedemment saisie a disparu. L'utilisateur peut penser que sa config a ete perdue.
- **Correction** : Ajouter un `console.warn` dans le catch de `loadFirebaseConfig()`. Si une config existait en IndexedDB mais n'est plus lisible, afficher un toast d'avertissement apres le boot.
- **Statut** : [x] corrige - console.warn ajoute dans le catch IndexedDB

### [RECOMMANDE] Accessibilite : labels de formulaire sans association explicite
- **Fichier** : `app.html:834-841` (setup), et formulaires dynamiques
- **Description** : Les labels du setup screen encapsulent les inputs (pattern valide) mais n'ont pas d'attribut `for`/`id` explicite. C'est acceptable pour le HTML statique, mais les formulaires generes dynamiquement en JS (charge, carburant, lavage) utilisent des `innerHTML` avec `<label>` et `<input id="...">` sans `for` correspondant. Les modales ont des `aria-labelledby` et `role="dialog"` corrects.
- **Correction** : Ajouter `for="..."` sur les labels des formulaires statiques ou verifier que les labels englobent bien les inputs.
- **Statut** : [x] corrige - for= ajoute sur tous les formulaires (entretien, echeances, carnet, lavage, carburant, charge)

### [RECOMMANDE] Nommage incoherent : melange francais/anglais dans les fonctions
- **Fichier** : `app.html` (global)
- **Description** : Le code melange nommage francais (variables : `echeance`, `lavage`, textes UI) et anglais (fonctions : `renderWash`, `addDeadline`, `getStatus`). C'est acceptable pour un projet perso, mais certains cas pretent a confusion : `renderFuel` vs section affichee "Carburant", `renderCharge` vs "Charges". Coherent dans l'ensemble mais le melange augmente la charge cognitive.
- **Correction** : Pas critique, mais si un refactoring est prevu, harmoniser vers l'anglais pour les noms de fonctions.
- **Statut** : [ ] ouvert

### [RECOMMANDE] Performance : renderAllSections() rend toutes les sections inutilement
- **Fichier** : `app.html:5140-5156`
- **Description** : `renderAllSections()` appelle tous les renderers de toutes les sections, meme celles invisibles. Elle est appelee lors du login, switch vehicule, import et sync. Pour la sync Firestore (onSnapshot), seule la section active devrait suffire dans la plupart des cas.
- **Correction** : Considerer l'utilisation de `renderAll(getActiveSection())` dans `onSnapshot` au lieu de `renderAllSections()`, sauf au login initial.
- **Statut** : [ ] ouvert

### [RECOMMANDE] Pas de validation format dans le setup Firebase
- **Fichier** : `app.html:1441-1465`
- **Description** : `submitSetup()` verifie que les champs requis ne sont pas vides mais ne valide pas le format (apiKey devrait commencer par "AIza", authDomain devrait contenir ".firebaseapp.com" ou ".web.app", projectId format alphanumerique). Un utilisateur qui copie mal sa config ne recevra qu'un "Erreur d'initialisation Firebase" generique.
- **Correction** : Ajouter des validations regex basiques avec des messages d'erreur specifiques par champ.
- **Statut** : [x] corrige - validation apiKey (AIza), authDomain (.firebaseapp.com/.web.app), projectId (alphanum+tirets)

### [RECOMMANDE] resetFirebaseConfig utilise confirm() - pas de confirmation visuelle post-action
- **Fichier** : `app.html:1467-1471`
- **Description** : `resetFirebaseConfig()` utilise `confirm()` natif puis `location.reload()`. Le `confirm()` est suffisant comme garde, mais l'utilisateur ne recoit pas de toast ou feedback avant le reload. Le reload est quasi-immediat donc c'est mineur, mais dans un contexte lent (offline), l'attente serait sans feedback.
- **Correction** : Acceptable en l'etat. Si souhaite, ajouter un toast "Configuration reinitialise" avant le reload avec un court delai.
- **Statut** : [ ] ouvert

### [RECOMMANDE] Duplication logique conso charges entre renderStats et computeChargeConsumption
- **Fichier** : `app.html:4314-4325` vs `app.html:1956-1988`
- **Description** : `renderStats()` recalcule la consommation kWh/100km des charges en full-to-full avec une boucle manuelle, alors que `computeChargeConsumption()` fait exactement le meme calcul. C'est du code duplique qui pourrait diverger.
- **Correction** : Utiliser `computeChargeConsumption(chargeEntries).realPoints` dans `renderStats()` pour obtenir les donnees de consommation au lieu de recalculer.
- **Statut** : [x] corrige - renderStats utilise computeChargeConsumption().realPoints

### [RECOMMANDE] GARAGE_DOC initialise a 'garages/main' alors que v3 n'utilise plus ce path
- **Fichier** : `app.html:1422`
- **Description** : `GARAGE_DOC` est initialise a `'garages/main'` avec un commentaire "updated to garages/{uid} after auth". Si du code s'execute avant l'auth et utilise `GARAGE_DOC`, il pointerait vers un mauvais document. En pratique, `saveStore()` verifie `currentUser` avant d'ecrire, mais la valeur par defaut est trompeuse.
- **Correction** : Initialiser a `null` ou `''` et ajouter un guard dans `saveStore` et `startFirestoreSync` qui verifie que `GARAGE_DOC` est bien set.
- **Statut** : [x] corrige - GARAGE_DOC initialise a null

### [RECOMMANDE] config.example.js mentionne ALLOWED_UIDS avec note retrocompat mais aucun code ne l'utilise
- **Fichier** : `config.example.js:16`, `app.html:20`
- **Description** : `ALLOWED_UIDS` est declare dans app.html (ligne 20) et dans config.example.js mais n'est jamais utilise fonctionnellement dans le code v3. Le commentaire dans config.example.js mentionne "conserve pour retrocompatibilite" mais il n'y a pas de code de retrocompatibilite qui le lit.
- **Correction** : Supprimer la declaration `ALLOWED_UIDS` de app.html ligne 20 et ajouter un commentaire plus clair dans config.example.js ou le retirer egalement.
- **Statut** : [x] corrige - ALLOWED_UIDS supprime de app.html et config.example.js
